Preview HTML

VS Code Extension — Idea Document

Problem Statement

How might we make viewing AI-generated HTML documents in VS Code as frictionless as reading a rendered page — preview-first, always current, no extra clicks?

Recommended Direction

Custom Editor (preview-first) with toggle to source

Register a CustomTextEditorProvider as the default editor for .html files. When a user opens any HTML file, they see the fully rendered preview — not raw markup.

A toolbar button in the editor title bar lets them toggle to the standard text editor when they need to inspect or edit source, and toggle back to preview with another click.

This inverts the default paradigm: every other HTML preview extension treats source as primary. For AI-generated documentation, the rendered view is what you care about. Source is the escape hatch.

Key Assumptions to Validate

  • Custom Editor coexistence — Users can seamlessly toggle between the custom editor and the built-in text editor via vscode.openWith.
    High confidence — this is a supported API pattern.
  • File watcher responsiveness — When an external process writes to the HTML file, the preview updates fast enough to feel "live".
    Test with rapid multi-save scenarios.
  • Relative asset resolution — Images, CSS, and JS with relative paths resolve correctly via webview.asWebviewUri().
    Needs testing with nested folder structures.
  • Users rarely edit these files — The toggle overhead (one click) is acceptable because the primary use case is reading rendered output.

Technical Approach

  • CustomTextEditorProvider with retainContextWhenHidden: true
  • vscode.openWith(uri, 'default') for toggling to text editor
  • Register with priority: "default" to win for .html files
  • File change detection via the TextDocument provided by Custom Editor API

Not Doing (and Why)

Embedded Monaco source editor inside the webview

Adds massive complexity (losing native extensions, keybindings, language services) for marginal gain.

Local dev server / proxy mode

Out of scope for v1. This is for static HTML documents, not running apps.

Multi-file dashboard / gallery view

Cool for later, but the single-file preview is the core value.

Visual diff between versions

Interesting for agentic workflows but adds significant complexity.

Script execution sandboxing (CSP)

For v1, allow scripts to run. Revisit if security concerns arise from real usage.

Open Questions

  1. Register as default for all .html files, or only within specific folders?
  2. Should there be a "pin" behavior (like Markdown preview's lock icon)?
  3. How to handle HTML files that are clearly application code (Angular templates, etc.)?
  4. Should the extension contribute a "Preview HTML" command to the palette?
